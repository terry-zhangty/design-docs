1 Background

Problem

Cursor Agent 缺乏稳定的长期记忆与项目级全局视野；业务逻辑与技术知识分散在 Obsidian 笔记与 Agent 对话历史中，难以被机器一致地检索与推理，并存在对话冷启动问题。

Goal

通过 MemOS 统一 API 与 Cursor MCP 协议，建立三层记忆模型，将碎片知识转化为可检索、可演进的知识与经验层，并与 Cursor 规则形成「先回忆、再回答、后归档」的闭环。

2 Architecture

三层模型定义：数据源、机制与职责一览。

Three-layer memory model
层级 名称 数据源 核心机制
Level 1 事实层 (Fact Layer) Obsidian 笔记 & Agent Transcripts 主动推送 (Push)：保持全库视野，解决冷启动。
Level 2 情境层 (Episodic Layer) 精炼后的对话历史 蒸馏推送 (Distillation)：下周开放问题。
Level 3 技能层 (Procedural Layer) SOP / Skills 经验进化 (Evolution)：下周开放问题。

2.1 Level 1：事实层

设计要点

事实层作为 Agent 的「第二大脑」，采用主动推送模式，使 MemOS 在对话开始前即具备相关知识覆盖。

实现：memos_sync.py

路径：experimental/tianyi.zhang/memos/memos_sync.py

  • 并发同步：ThreadPoolExecutor（10 线程），批量推送至秒级。
  • 接口：MemOS .../v1/add/message；支持 .md 与 .jsonl。
  • 范围：my_obsidian_vault、my_agent_transcripts。
  • Git Hook（计划中）：在 vault-bridge/sync.sh 增加 after_sync，调用 memos_sync.py --incremental。

2.2 Cursor 交互：.cursor/rules/memos.mdc

检索协议 (Recall Protocol)

通过 user-memos-api-mcp 强制执行「先回忆 → 再回答 → 归档」闭环：

  1. 自动检索（回答前 search_memory）
    • query：当前消息摘要
    • relativity：0.7
    • knowledgebase_ids：["all"]
    • conversation_first_message：首轮用户消息
  2. 安全判定：四步判定法（来源验证、主体确认、关联校验、时效检查），与规则全文一致。
  3. 自动归档：回答后 add_message 同步当前 Turn。
