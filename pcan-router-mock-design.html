核心结论

Link-time Mock

处理闭源驱动（如供应商 .so）的成熟手段：在链接阶段替换头文件对应实现，隔离硬件依赖。

Pluggable Backends

行业标准：同一套驱动代码支持多种后端。

  • 内存后端 — CI 确定性测试，无网络开销。
  • 中间件后端 — SIL/HIL，分布式与闭环控制。

Side-channel Injection

测试钩子可加速单测；发布版本必须严格隔离，避免注入面泄漏到生产路径。

行业标准对标（NVIDIA / Vector / ROS 2 / Autoware）

对比各方案的做法与价值，用于交叉验证 PCAN Router 的 Mock 分层策略。

方案 / 工具 做法 价值
NVIDIA DriveWorks 虚拟传感器 (Virtual Sensors) / 回放接口 同一套 API 支持真实硬件与文件回放，保证应用逻辑一致性。
Vector SIL Kit 仿真总线 (Simulation Bus) 中间件连接虚拟 ECU 与仿真引擎，支持复杂总线协议协同仿真。
ROS 2 mock_components 硬件接口 Mock 实现（ros2_control 插件） 不带硬件即可测试控制器逻辑。
Autoware Vehicle Interface 抽象 底层车辆通信隐藏在 ROS 接口之后，以不同 package 适配硬件/仿真。

PCAN Router Driver — 设计与验证要点

Link-time Mocking

建议通过 Bazel 链接配置在 can_router_real 与 can_router_mock 之间切换 — 与调研结论一致。

分层注入

  • CI：内存队列 + Mock_PushMcuFrame()，追求速度与确定性。
  • 仿真：中间件（如 DDS）中转，对接外部仿真（Carla 等）。

数据形态：Raw CanFrame → Pre-packaged Payload

将上层从「逐帧原始 CAN」推进到「已打包、可版本化的载荷」，便于 Mock 与真实路径共享同一语义层，减少 Dual Reality。

RAW 底层 CanFrame / 字节流
→
SEMANTIC 路由 / 聚合 / 校验
→
PAYLOAD 预打包 MCU / 业务载荷

故障注入：Action Sequence

用可编排的动作序列（延迟、丢帧、错误 DLC、断链重连等）驱动 Mock 后端，而非散落的全局标志位 — 与 Side-channel 测试钩子配合，但需在 Release 构建中剔除。

阶段 动作示例 目的
Priming 连接建立、版本握手 验证状态机与超时
Steady 周期帧、正常载荷 回归与性能基线
Fault 注入序列：延迟 → 突发丢帧 → 恢复 重连、缓冲与告警路径

潜在风险与缓解

保留 Nightly 集成测试覆盖真实 I/O 路径，避免 Mock 绿、实车红。
在 CI 中加入 static_assert 或偏移量检查，监控 Mock 与 Real 结构体一致性。
