Mock Control CLI Tool — Option Comparison

Two approaches for sending MockControlTrigger commands without the APOV UI
Recommended: Build A first, then extend to B — they share the same core

Start Here

Phase 1 — One-Shot Tool

Run once, send one command, exit. Chain in scripts. Build first.

Build Time~1 hour
Code Size~150–200 lines
ComplexityLow
InteractivityNone
ScriptabilityExcellent
Live FeedbackNo

Pros

  • Fast to implement
  • Highly scriptable & composable
  • Simple codebase, easy to maintain
  • Follows existing codebase patterns
  • Great for CI/CD pipelines

Cons

  • No real-time status feedback
  • DDS startup overhead per invocation
  • Multiple runs needed for sequences

Usage Example

# Switch driving mode to auto
mock_ctl --mode=auto

# Start throttle signal at 50% with step profile
mock_ctl --start --throttle=0.5 --profile=step

# Stop all signals
mock_ctl --stop

# Scripted sequence
mock_ctl --mode=auto && sleep 2 && \
mock_ctl --start --throttle=0.5

Recommended: Incremental Build Strategy

Option B builds on top of Option A — you don't have to choose one or the other

The core send logic is identical between both options. The only difference is the entry point. Build Option A first for immediate value, then extend to Option B later if interactive debugging proves valuable. The shared core means upgrading from A to B costs only ~1–2 additional hours instead of 2–3 from scratch.

Phase 1: Build Option A — One-Shot Tool

~1 hour • Immediately usable
  • Create shared library: AOS node, MockControlTrigger builder, DDS publisher
  • Build mock_ctl binary with CLI flags
  • Test with onboard_main in mock control mode
  • Use in bash scripts and CI/CD pipelines

Phase 2 (optional): Extend to Option B — REPL

~1–2 additional hours • Reuses Phase 1 core
  • Add REPL loop: read command → parse → call shared send logic
  • Add DDS subscriber for mock_control_status on background thread
  • Add live status display formatting
  • Build interactive_mock_ctl binary

Code Structure (Shared Core)

mock_control_cli/
  BUILD                          # Bazel build targets
  mock_control_sender.h          # Core logic (shared by A and B)
  mock_control_sender.cc         # AOS node, message builder, publisher
  mock_ctl_main.cc               # Phase 1: one-shot entry point
  interactive_mock_ctl_main.cc   # Phase 2: REPL entry point (added later)
