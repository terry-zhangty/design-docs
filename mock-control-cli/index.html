Mock Control CLI Tool — Option Comparison

Two approaches for sending MockControlTrigger commands without the APOV UI

Recommended

Option A — One-Shot Tool

Run once, send one command, exit. Chain in scripts.

Build Time~1 hour
Code Size~150–200 lines
ComplexityLow
InteractivityNone
ScriptabilityExcellent
Live FeedbackNo

Pros

  • Fast to implement
  • Highly scriptable & composable
  • Simple codebase, easy to maintain
  • Follows existing codebase patterns
  • Great for CI/CD pipelines

Cons

  • No real-time status feedback
  • DDS startup overhead per invocation
  • Multiple runs needed for sequences

Usage Example

# Switch driving mode to auto
mock_ctl --mode=auto

# Start throttle signal at 50% with step profile
mock_ctl --start --throttle=0.5 --profile=step

# Stop all signals
mock_ctl --stop

# Scripted sequence
mock_ctl --mode=auto && sleep 2 && \
mock_ctl --start --throttle=0.5

Option B — Interactive REPL

Persistent process with live status and command loop.

Build Time~2–3 hours
Code Size~400–500 lines
ComplexityMedium
InteractivityFull REPL
ScriptabilityModerate
Live FeedbackYes

Pros

  • Interactive and intuitive UX
  • Real-time actuator status display
  • Excellent debugging experience
  • Single persistent process
  • Tab completion potential

Cons

  • More complex implementation
  • Harder to script (pipe commands)
  • Command parser adds surface area
  • More code to maintain

Usage Example

$ mock_ctl_interactive
Mock Control CLI v1.0
Connected to mock_control_trigger topic.
Listening on mock_control_status...

> mode auto

> status
  Throttle: 0.50 [ACTIVE]
  Brake: 0.00 [IDLE]
  Steer: 0.00 [IDLE]

> stop

> quit

  │   mock_ctl binary     │
  │  (flags: --mode,     │
  │   --start, --stop)   │
  └──────────┬───────────┘
             │ publish
             ▼
  ┌──────────────────────┐
  │  AOS/DDS Topic       │
  │ mock_control_trigger │
  └──────────┬───────────┘
             │ subscribe
             ▼
  ┌──────────────────────┐
  │  MockControl         │
