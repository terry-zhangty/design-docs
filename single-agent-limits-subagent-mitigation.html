单代理限制与子代理缓解

结构性问题，与 prompt 写法关系不大。

开场 · 30s

规律性的现象

  • 做到一半开始 hallucinate 函数签名。
  • 拿到 "done"，打开一看根本跑不了。
  • 长 session 越到后面越不听话，开始动没让它动的东西。

起初以为是 prompt 的问题。后来发现不是——这是 单代理架构的结构性问题，跟怎么写 prompt 关系不大。

根因 · 60s

子代理 · 60s

不是更大的模型，是不同的架构

更大的 context window 解决不了 U 形注意力——这是 attention 机制本身的特性，不是规模问题。

Context 隔离 独立验证 范围约束

  • Context 隔离：子代理的 grep、中间状态不污染主 context；父代理只消费压缩摘要——等价于函数边界。
  • 独立验证：实现与验证在不同 context；验证者只拿到 artifact——code review 不能自己 review 自己。
  • 范围约束：self-contained prompt，无 15 轮历史；做完即销毁，不会漂移。
专业化 agent vs 通用 agent
+2.75% 完成率
token 成本
−96%

成本降幅比完成率提升大两个数量级——大部分 token 消耗在无效的 context 上。
